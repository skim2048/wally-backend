# wally — SRS 작성 배경 및 합의 사항

> 본 문서는 SRS 본문이 아니다. SRS 작성에 착수하기 전, 지금까지의 논의에서
> 확정된 판단과 작업 방식을 한곳에 정리해 둔 기록이다. 이후 SRS의 각 절은
> 이 문서의 결정을 출발점으로 삼는다.

---

## 1. 프로젝트 개요

- **babycat** — VLM을 사용하는 범용 백엔드. 특정 용도에 묶이지 않은 기반 시스템이다.
  RTSP 카메라 영상을 받아 일정 간격으로 프레임을 VLM에 넣어 추론하고,
  사용자가 지정한 트리거 키워드가 매칭되면 그 순간 전후를 클립으로 저장·알림한다.
- **wally** — babycat을 기반으로 `반려견 건강관리를 위한 AI 펫하우스`라는
  단일 용도로 확장 개발하는 전용 프로젝트.
- 개발 방식 — babycat 베이스 코드를 복제해 쓰는 편이 편리하나, babycat은
  프로토타입이라 설계적 결함이나 임시 패치 코드가 있을 가능성이 높다.
  따라서 **방향성(VLM 기반이라는 큰 틀)은 유지하되, SRS를 작성하면서
  밑바닥부터 점진적으로 다시 설계**한다. SRS 작성이 곧 재설계의 수단이다.

## 2. 선행 설계 논의에서 확정된 판단

아래는 SRS 착수 이전의 설계 대화(`tmp/conversation.md`, `tmp/hand-off.md`,
`tmp/draft.md`)에서 확정된 내용의 요약이다. 상세 근거는 해당 원본을 참조한다.

### 2.1 포지셔닝

- wally는 **반려견 웰니스(wellness)** 제품이며, 의료기기가 아니다.
- 따라서 출력 문구는 다음 세 가지를 준수해야 한다.
  1. 질병명을 말하지 않는다 — "관절염 의심"은 안 되고, "뒤척임이 평소보다 30% 많음"은 된다.
  2. 행동을 권고하지 않는다 — "즉시 수의사 진료가 필요하다"는 안 된다.
  3. 절대 기준으로 판정하지 않고, 개체의 평소 상태 대비 변화 추세만 말한다.
     "정상 호흡수는 30회 이하"는 안 되고, "평소보다 늘었다"는 된다.
- 이 세 가지는 SRS에서 검증 가능한 제약·규칙으로 명문화한다.

### 2.2 물리·감지 제약

- **개방형 하우스** — 입구가 열린 물리적 하우스. 밀폐형 아님.
- **비접촉식** — 개체에 아무것도 부착하지 않음.
- 위 두 제약의 귀결: 공간 기반 센서(온습도·가스)는 개방 구조에서 변별력이
  낮으므로 배제. 유효 신호는 **개체 자체에서 나오는 영상**으로 수렴.

### 2.3 핵심 구조

- babycat의 방식: 단일 시점 키워드 매칭 후 즉시 클립 녹화·알림. 즉각적이나
  지속성과 신뢰도가 낮다.
- wally의 방식: VLM 추론을 일정 간격으로 기록하고, 주기적으로 빈도를 합산하고
  패턴을 요약한다.
- 신뢰도 근거: 단일 추론이 부정확해도 누적 빈도는 참값에 수렴한다. 화각이
  고정되면 VLM의 편향이 시간에 걸쳐 상수가 되어 "이번 주 − 지난주" 차분에서
  소거된다. 따라서 검증 대상은 "VLM이 정확한가"가 아니라
  **"편향이 시간에 걸쳐 안정적인가"**로 좁혀진다.
- 설계 원칙: 단일 프레임 판정 금지, **개체의 평소 상태 대비 변화 추세로만 출력**.
  이 처방은 2.1의 규제 기준 3(상대 변화만 본다)과 동일하다.
  기술적 약점 보완책이 곧 규제 회피책이다.
- 유의점: 누적을 요약하는 단계에서 진단이 재유입되기 쉽다. 요약은 빈도와
  변화 서술까지만 수행하도록 제한해야 한다.

### 2.4 추적 대상 지표와 우선순위

- 지표 선정 기준은 두 가지를 상정할 수 있다. 하나는 그 지표가 실제 건강과
  얼마나 깊이 연관되는가이며, 다른 하나는 구현 결과가 얼마나 분명하게
  드러나는가이다. 점진적 검증 및 개선을 위해 후자(구현 결과의 가시성)를 우선한다.
- 지표 우선순위:
  1. **공간 점유** — 개체의 하우스 내 존재 여부. 구현이 가장 용이하고 나머지 관찰의 선행 조건.
  2. **자세 분포** — 눕기/웅크리기/앉기 등의 비율. VLM의 강점 영역이자 주력 지표.
  3. **행동 빈도** — 8가지 관심 행동(선회·긁음·구토/헛구역질·핥음·헐떡임·짖음/하울링·기침·발작)의
     발생 횟수. 수일간 축적되어야 의미가 형성되므로 후순위.
- 미결정 사항: 자세 분포가 지표로서 성립하려면, VLM의 출력이 매 추론마다
  미리 정의된 자세들 중 하나로 일관되게 귀착되어야 한다. 그러나 VLM은 분류
  모델이 아니라 생성 모델이므로, 같은 자세도 표현이 갈린다(예: 선회를
  'circling'으로도 'walking around'로도 출력). 따라서 (가) 자세 범주의 개수와
  정의, (나) 자유 생성된 표현을 그 범주로 모으는 방법(프롬프트 제약, 제약
  디코딩, 사후 정규화 등)이 모두 아직 정해지지 않았다.

### 2.5 펫하우스라는 공간의 타당성

- 행동 관점(휴식 공간): 정적·소규모 행동만 관찰 가능 → 공간 제약에 의한 약점.
- 관찰 관점(고정된 배경·화각·지속 관찰): 성능과 신뢰도를 높이는 핵심 강점.
- 보안 관점(사람 생활공간과 비중첩): 프라이버시를 보장하는 제품 정당성 근거.
- 건강 관리를 포착해야 할 사건이 아니라 유지되는 평소 상태로 재정의하면,
  관찰 가능 행동이 좁다는 한계가 오히려 안정적인 평소 상태 확립의 전제가 된다.
  행동 관점의 약점과 관찰 관점의 강점이 동일한 사실의 두 측면으로 수렴한다.

## 3. babycat 기반 시스템의 현 구조 (재설계의 출발 코드)

`../babycat`의 코드 기반 요약(`SUMMARY.md`)에서 확인된 골격이다. wally는 이
구조를 기준선으로 삼아 유지/변경/추가를 판단한다. 여기서 wally는 엄밀히 따지자면
wally-backend를 가리키며, wally-frontend는 추후 추가할 예정이다.

- **3 컨테이너**: `mtx`(MediaMTX 미디어 서버) / `app`(GStreamer + VLM +
  stdlib HTTP 서버 :8080) / `api`(FastAPI 게이트웨이 :8000). 외부 평면(api)과
  내부 평면(app)이 같은 JWT 시크릿을 공유한다.
- **파이프라인**: 카메라 → mtx → app이 RTSP를 1fps로 받아 RingBuffer →
  최근 N프레임을 VLM에 투입 → 키워드 매칭 시 클립 저장.
- **VLM 자식 프로세스 격리**: 성능이 아니라 CUDA/TVM 메모리 누수 회피 설계.
  모델 전환 = 자식 종료 후 재spawn.
- **트리거 클립 rollover 모드**: tmpfs에 1초 단위 세그먼트를 항상 기록해
  사후 인지 시 사전 영상까지 회수.
- **하드웨어**: NVIDIA Jetson Orin NX 16GB(JetPack 6.2), RTSP H.264 IP 카메라(ONVIF PTZ 선택).
- **VLM**: NanoLLM + VILA1.5-3b(MLC, INT4).

→ wally의 핵심은 babycat 위에 한 계층을 추가하는 데 있다. 범용 키워드 매칭에는 babycat의
  코드를 그대로 사용하고, 그 위에 추론 스트림을 시계열로 누적하여 주기적으로 집계·요약하는
  **매크로 계층**을 얹는다.

## 4. SRS 작업 방식 합의

### 4.1 템플릿

- 사용자가 보유한 SRS 서적의 템플릿을 **기준 골격으로 채택**한다. 이는 IEEE 830의
  상위 집합으로, 환경(3장)·변경관리(8장)·승인(9장) 등 운영·프로세스 절을 더한 구성이다.
- 골격 구조(9개 장):
  1. introduction
  2. overall description
  3. environment
  4. external interface requirements
  5. performance requirements
  6. non-functional requirements
  7. functional requirements
  8. change management process
  9. document approvals

### 4.2 작성 원칙

- **구조는 고정, 내용 수준은 점진 결정.** 골격은 위 템플릿으로 못 박되, 각 절을
  실질·상속·해당없음 중 무엇으로 채울지는 작성이 진행되며 정한다. 절의 충전 수준은
  프로젝트 규모·성격에 따라 달라지며, 그 성격은 scope를 확정한 뒤에야 드러나기 때문이다.
- 각 절의 성격 구분(잠정):
  - **실질** — 결정이 필요한 핵심. 2.1 product perspective, 2.8 backward compatibility,
    2.3 overall operation, 2.4 project functions, 5 performance, 6.5/6.6/6.13(규제·제약),
    7 functional requirements 등.
  - **상속** — babycat에서 거의 그대로 이어받는 절. 3 environment, 4 external interface,
    6.2 security 등.
  - **해당없음/추후** — 이 단계에서 형식만 두고 비워둘 절. 3.6 configuration management,
    3.7 bugtrack, 6.11 unicode, 6.12 64bit, 8 change management, 9 approvals 등.
- 좋은 요구사항의 속성을 준수한다: 검증 가능, 명확, 추적 가능(ID 부여), 구현 비종속.
